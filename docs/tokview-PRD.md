# tokview — PRD (Product Requirements Document)

> 여러 AI 에이전트(Claude Code, Hermes 등)를 하나의 화면에서 운용하고,
> 활성 터미널의 토큰 사용량을 실시간으로 확인하는 터미널 작업 환경(TUI).

- 문서 버전: v1.0
- 작성일: 2026-06-04
- 대상 플랫폼: macOS (1차), Linux (확장)
- 프로젝트 경로: `/Users/jdethankim/ClaudeCode/tokview`

---

## 1. 개요

tokview는 Warp나 Zellij처럼 **여러 터미널 세션을 하나의 UI에서 관리**하되,
각 세션에서 실행 중인 AI 에이전트의 **토큰 사용량을 곁에서 실시간으로 보여주는** 데 특화된
터미널 작업 환경이다.

핵심 화면은 3단 구조다.

1. 왼쪽: 열려 있는 터미널 세션 목록(사이드바). 어떤 에이전트가 어느 상태인지 확인하고 전환한다.
2. 가운데: 선택된 터미널 세션이 실제로 살아 있는 상태로 표시되는 메인 작업창.
3. 오른쪽: 가운데에서 활성화된 세션에 연동된 토큰 사용량 패널.

사이드바에서 세션을 고르면 가운데와 오른쪽이 그에 맞춰 자동으로 전환되는 것이 핵심 동작이다.

---

## 2. 문제 정의

- AI 에이전트(Claude Code, Hermes 등)를 여러 개 동시에 돌릴 때, 각각을 별도 터미널 창으로 띄워
  창 사이를 오가야 해서 관리가 번거롭다.
- 작업 중 토큰을 얼마나 쓰고 있는지 한눈에 보이지 않는다. 사용량 확인은 보통 별도 명령어
  (`ccusage` 등)를 따로 실행해야 한다.
- 기존 토큰 모니터 도구들은 "작업 화면"과 "사용량 화면"이 분리돼 있어, 동시에 보기 어렵다.

tokview는 이 세 가지를 하나의 화면으로 합친다.

---

## 3. 목표와 비목표

### 3.1 목표 (Goals)

- 하나의 UI에서 여러 터미널 세션을 만들고, 목록에서 보고, 전환한다.
- 가운데 패널에서 선택된 세션의 에이전트를 실제로 입력·실행·응답까지 사용할 수 있다.
- 활성 세션의 토큰 사용량을 오른쪽 패널에 실시간(또는 수 초 주기 폴링)으로 보여준다.
- 세션별로 어떤 에이전트가 어떤 상태(실행 중 / 대기 / 완료)인지 사이드바에서 식별한다.
- 설정 파일로 레이아웃, 폴링 주기, 테마 등을 조정할 수 있다.

### 3.2 비목표 (Non-goals)

> 범위를 명확히 하기 위해, **이번 프로젝트에서 하지 않는 것**을 분명히 적는다.

- Warp를 완전히 대체하는 범용 터미널 에뮬레이터를 만들지 않는다. tokview는 AI 에이전트 운용에
  특화된 작업 환경이다.
- 새로운 셸(zsh/bash 대체)을 만들지 않는다.
- Claude Code / Hermes 자체를 다시 구현하지 않는다. tokview는 이들을 호스팅해서 보여줄 뿐이다.
- 1차 버전에서 Windows 네이티브 지원은 목표가 아니다(macOS 우선).

---

## 4. 사용자 / 페르소나

- 1인 개발자 / 소규모 팀에서 여러 AI 에이전트를 병렬로 돌리며 작업하는 사람.
- 토큰/비용 소모에 민감하고, 작업 중 사용량을 곁눈으로 확인하고 싶은 사람.
- 터미널 환경에 익숙하며, GUI 앱보다 키보드 중심 작업을 선호하는 사람.

---

## 5. 핵심 기능 (우선순위별)

우선순위는 P0(필수) > P1(중요) > P2(선택/실험)으로 표기한다.

### P0 — 반드시 동작해야 하는 것

- **PTY 기반 라이브 터미널 (가장 어려운 핵심)**
  가상 터미널(PTY)로 에이전트를 띄우고, 그 화면을 가운데 패널에 실시간 렌더링한다.
  키 입력을 해당 PTY로 전달하고, 출력(이스케이프 시퀀스 포함)을 파싱해 화면에 그린다.
  이 부분이 동작하지 않으면 나머지가 의미 없으므로 가장 먼저 검증한다.

- **3단 정적 레이아웃**
  사이드바 / 메인 / 사용량 패널의 3단 구조를 화면에 배치한다.

- **세션 생성 및 전환**
  새 터미널 세션을 만들고, 사이드바 목록에서 선택하면 가운데가 그 세션으로 전환된다.

### P1 — 제품의 핵심 가치

- **활성 세션 연동 사용량 패널**
  가운데에서 활성화된 세션의 에이전트 종류에 맞춰, 오른쪽 패널이 해당 사용량을 보여준다.
  `ccusage`의 JSON 출력을 주기적으로 호출해 직접 렌더링한다.
  (claude → 실시간에 가깝게, hermes → 수 초 주기 폴링)

- **세션 상태 표시**
  사이드바에 각 세션의 에이전트 이름과 상태(실행 중 / 대기 / 완료)를 표시한다.

- **세션별 사용량 귀속**
  각 세션을 서로 다른 작업 디렉터리(프로젝트)에서 실행하고, `ccusage`의 프로젝트/세션 단위
  집계를 활용해 세션별 사용량을 구분해서 보여준다.
  (주의: ccusage는 전역 로그를 읽으므로, 패널 단위가 아니라 프로젝트/세션 단위로 귀속된다 — 8장 참고)

### P2 — 선택 / 다듬기

- 테마 / 색상 커스터마이징, 키맵 재정의, 레이아웃 비율 조정.

## 6. 사용자 시나리오

### 시나리오 A — 기본 운용

1. 사용자가 tokview를 실행한다.
2. 사이드바에 기본 세션 목록이 보인다(없으면 빈 상태 + "세션 추가" 안내).
3. 사용자가 새 세션을 만들고 `claude`를 선택한다.
4. 가운데 패널에 Claude Code가 실제로 실행되어 입력 가능한 상태가 된다.
5. 오른쪽 패널에 이 세션(claude)의 토큰 사용량이 나타난다.
6. 사용자가 또 다른 세션을 만들어 `hermes`를 띄운다.
7. 사이드바에서 두 세션을 오가면, 가운데와 오른쪽이 선택에 맞춰 전환된다.

### 시나리오 B — 사용량 확인

1. claude 세션에서 작업을 진행한다.
2. 에이전트가 응답할 때마다 오른쪽 패널의 누적 토큰/비용이 갱신된다(폴링 주기 내).
3. 사용자는 창 전환 없이 작업과 사용량을 동시에 본다.

## 7. 기술 스택 & 아키텍처

### 7.1 기술 스택

- 언어: Python 3.14 (현재 환경 기준)
- TUI 프레임워크: Textual
- 터미널 에뮬레이션: `pty`(표준 라이브러리) + `pyte`(이스케이프 시퀀스를 화면 버퍼로 파싱)
- 사용량 데이터: `ccusage` (외부 CLI, JSON 출력 사용)
- 패키지 격리: `.venv`

### 7.2 아키텍처 개요

```
┌────────────────────────────────────────────────────────────┐
│                         tokview (Textual App)               │
│                                                             │
│  ┌────────────┐   ┌──────────────────┐   ┌──────────────┐   │
│  │  Sidebar   │   │   Terminal View   │   │  Usage Panel │   │
│  │ (세션 목록)  │──▶│ (활성 PTY 렌더링)   │──▶│ (활성 사용량)  │   │
│  └────────────┘   └──────────────────┘   └──────────────┘   │
│        │                   ▲                     ▲           │
│        │ 선택              │ 화면 버퍼            │ 폴링 결과    │
│        ▼                   │                     │           │
│  ┌──────────────────────────────────────────────────────┐  │
│  │              SessionManager (상태의 단일 출처)            │  │
│  │  - 세션 목록 / 활성 세션 ID                                │  │
│  │  - 세션별: PTY, pyte 화면, 에이전트 종류, 작업 디렉터리       │  │
│  └──────────────────────────────────────────────────────┘  │
│        │                                     │              │
│        ▼                                     ▼              │
│   PTY 프로세스(claude/hermes ...)        UsagePoller          │
│   os.openpty + subprocess              (ccusage --json 주기 호출)│
└────────────────────────────────────────────────────────────┘
```

### 7.3 핵심 동작 흐름

- 키 입력: Textual → 활성 세션의 PTY로 write.
- 화면 출력: PTY read → pyte로 파싱 → 화면 버퍼 → Terminal View가 렌더링.
- 세션 전환: Sidebar 선택 → SessionManager의 활성 세션 변경 → Terminal View와 Usage Panel이
  새 활성 세션을 구독해 다시 렌더링.
- 사용량: UsagePoller가 활성 세션의 에이전트 종류/프로젝트에 맞는 `ccusage` 명령을 주기적으로
  호출 → JSON 파싱 → Usage Panel 갱신.

## 8. 기술적 제약 & 리스크 (정직한 한계)

> 이 장은 개발 중 "안 되는 이유"로 시간을 낭비하지 않도록, 알려진 어려움을 미리 못 박는다.

1. **PTY 렌더링이 최대 난관 (P0 리스크)**
   에이전트의 TUI는 색상, 커서 이동, 부분 갱신 등 복잡한 이스케이프 시퀀스를 쓴다.
   pyte로 대부분 처리되지만, 일부 고급 시퀀스나 리사이즈 처리에서 깨질 수 있다.
   → Phase 0에서 단일 터미널 렌더링을 먼저 검증하고 넘어간다.

2. **세션별 사용량 귀속의 한계**
   `ccusage`는 패널이 아니라 전역 로그(JSONL)를 읽는다. 따라서 "이 패널의 사용량"은
   엄밀히는 "이 에이전트 종류 / 이 프로젝트의 사용량"으로 귀속된다.
   세션마다 작업 디렉터리를 분리하고 프로젝트 단위 집계를 활용하는 것이 현실적인 방법이다.

3. **hermes 실시간성 한계**
   `ccusage`의 실시간 블록(`blocks --live`)은 Claude 전용 로컬 데이터에 의존한다.
   hermes는 주기 폴링(예: 5초)으로 갱신하며, claude만큼 매끄럽지 않다.

4. **토큰 수 정확도**
   글자 수 기반 추정은 부정확하다. 사용량은 `ccusage`가 읽는 실제 로그 수치를 기준으로 한다.

## 9. 마일스톤 (단계별 개발)

> 한 번에 전부 만들지 않고, 위험이 큰 부분부터 검증하며 쌓는다.

- **Phase 0 — PTY 개념 검증 (가장 먼저, 가장 중요)**
  Textual 위에 PTY 터미널 하나를 띄워, claude가 실제로 입력·응답되는지 검증.
  여기서 막히면 설계를 재검토한다.

- **Phase 1 — 3단 정적 레이아웃**
  사이드바 / 메인 / 사용량 패널의 빈 골격을 배치하고, 메인에 Phase 0의 터미널을 얹는다.

- **Phase 2 — 다중 세션 & 전환**
  여러 세션을 만들고 사이드바에서 선택 시 메인이 전환되도록 SessionManager 구현.

- **Phase 3 — 사용량 패널 연동**
  UsagePoller로 `ccusage --json`을 주기 호출, 활성 세션에 맞춰 사용량 패널 갱신.

- **Phase 4 — 다듬기**
  세션 상태 표시, 테마, 키맵, 레이아웃 비율, 설정 파일.

---

## 10. 성공 기준

- Phase 0: 가운데 패널에서 claude가 평소처럼 입력·응답된다.
- Phase 2: 세션 2개 이상을 만들고 끊김 없이 전환할 수 있다.
- Phase 3: 활성 세션을 바꾸면 오른쪽 사용량 패널이 그에 맞게 바뀐다.
- 전체: 창 전환 없이 한 화면에서 다중 에이전트 작업 + 사용량 확인이 가능하다.
